# Clinical Document Intelligence — Live Demo

**EMNLP 2026 System Demonstration**

A clinical document intelligence platform that extracts structured entities
from unstructured NHS clinical letters (referrals, clinic letters, discharge
summaries), detects safety-relevant flags (allergy conflicts, overdue
follow-ups, duplicate medications), and identifies contradictions across a
patient's document history.

This repository provides **two complementary ways to interact with the system**:

1. **A static viewer** with two pre-computed synthetic patients — no API
   keys, no cloud credentials, no external services required. Instant
   review.
2. **A standalone, credential-gated pipeline** (`pipeline/`) implementing
   the full extraction and four-guard grounding validation described in
   the paper — runnable live, on your own documents, with your own
   Anthropic API key.

## Option 1: Static viewer (no credentials needed)

Requires [Docker](https://www.docker.com/products/docker-desktop/) and
Docker Compose (bundled with Docker Desktop).

```bash
git clone https://github.com/H2908/clinical-document-intelligence-demo
cd clinical-document-intelligence-demo
docker-compose up
```

Wait ~30 seconds for both containers to start, then open:

**http://localhost:3001**

### What you'll see

Two synthetic patients, fully pre-processed through the pipeline:

- **Margaret Thompson** (patient_001) — cardiology/CKD case with a planted
  allergy contradiction (NKDA recorded in two documents, penicillin allergy
  recorded in a third). Demonstrates the contradiction-detection agent and
  the flag-identity matcher.
- **Daniel Ofori** (patient_002) — type 2 diabetes case with **zero**
  planted contradictions, used as a false-positive control to confirm the
  system doesn't hallucinate disagreements on a consistent chart.

Explore:
- **Briefing** — clinical summary card (conditions, medications, open flags)
- **Flags** — severity-tagged safety flags with source document linkage
- **Contradictions** — side-by-side conflicting claims with explanation
- **Timeline** — chronological event view, filterable by type
- **Documents** — the underlying synthetic clinical letters

## Option 2: Standalone pipeline (live inference, your own API key)

```bash
cd pipeline
python3.11 -m venv venv
source venv/bin/activate   # or venv\Scripts\activate on Windows
pip install -r requirements.txt
pip install https://s3-us-west-2.amazonaws.com/ai2-s2-scispacy/releases/v0.5.4/en_core_sci_sm-0.5.4.tar.gz
```

Create a `.env` file in `pipeline/` containing your own key:
```
ANTHROPIC_API_KEY=your-key-here
```

Then run:
```bash
python run_pipeline.py --input path/to/document.pdf --output results/output.json --verbose
```

Full setup, scope disclaimers, and test suite: see `pipeline/README.md`.
Verified via clean-install testing (fresh Python 3.11 virtual environment,
no dependency on any pre-existing environment): **39/39 tests pass**,
covering the paper's flagship composition-fabrication rejection example
and the calibration fix described in the paper's Evaluation section.

## Architecture

```
Static viewer:
  Browser (Next.js) --> FastAPI demo backend --> pre-computed JSON fixtures

Standalone pipeline (pipeline/):
  PDF --> extraction.py (NER + negation) --> llm_client.py (Claude Sonnet 4.6)
      --> grounding.py (four-guard validation) --> JSON output
```

The demo backend (`demo/api/main.py`) serves pre-computed pipeline outputs
from `demo/fixtures/*.json`. These fixtures were generated by running the
**full production pipeline** (scispaCy NER, rule-based flag detection,
LLM-assisted contradiction detection) against synthetic clinical documents,
then exporting the results. This means what you see in the static viewer is
real system output, not mocked data — just pre-computed so it requires no
external API keys or cloud services to run.

The standalone pipeline (`pipeline/`) is a deliberately narrower, publicly
runnable slice of the same underlying system: it implements extraction and
grounding validation faithfully (verified to produce an identical entity
count against the production pipeline on a real test document), but does
not include lab-observation parsing, full relative-date normalisation,
cross-document contradiction detection, or an OCR fallback for scanned
documents. See `pipeline/README.md` for the complete scope disclaimer.

## Performance

**Extraction pipeline** (measured on the source documents, not the demo
viewer layer): ~0.2-1.3 seconds per document for full NER + flag
extraction on commodity hardware.

**LLM-assisted stages** (live inference via `pipeline/run_pipeline.py`):
2-24 seconds per document (mean 10-12s), dependent on Anthropic API
latency.

**Static viewer resource footprint** (measured via `docker stats` at
idle): the API container uses ~35 MiB memory and under 0.4% CPU; the
frontend container uses ~67 MiB memory and under 0.4% CPU. This is
consistent with the viewer serving pre-computed fixture data rather than
performing live inference. Concurrent-request load has not been tested;
no throughput claim is made beyond this idle measurement.

See the paper for full evaluation details, including generalisation
testing against 30 real (non-synthetic) clinical documents.

## Repository structure

```
demo/
  api/main.py            - standalone FastAPI serving fixture data
  fixtures/*.json         - pre-computed pipeline outputs (2 patients)
  Dockerfile.api          - API container
frontend/                 - Next.js UI (shared with production system)
pipeline/
  extraction.py            - PDF -> typed, negated entities (standalone)
  grounding.py              - four-guard grounding validator (standalone)
  llm_client.py             - Claude Sonnet 4.6 wrapper
  run_pipeline.py           - end-to-end orchestration (PDF in, JSON out)
  demo.ipynb                - interactive walkthrough notebook
  tests/                    - 39 tests, pytest
  README.md                 - full setup, scope disclaimers
docker-compose.yml         - one-command static-viewer orchestration
```
